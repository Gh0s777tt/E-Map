**plan() nie sprawdza res.ok — odpowiedź 429/4xx z /api/route wywala render mapy (biały ekran)**

Realny trigger: rate-limit 429 (sliding window 30/60s w lib/ratelimit.ts) — a funkcja auto-reroute (#309, planRef.current → void plan(), l.1236) wywołuje `plan()` automatycznie przy zdarzeniach realtime, więc burst 429 jest osiągalny bez akcji użytkownika. Skutek kaskadowy: (1) `setResult(r)` (l.1261) ustawia `result` na obiekt błędu; (2) render `{result && <RouteSummary result={result}/>}` (l.1817) wchodzi, bo obiekt jest truthy; (3) RouteSummary czyta `result.segments.length` (mapPanels.tsx:57) na `undefined` → TypeError w RENDERZE. Dodatkowo synchronicznie w samym `plan()`: `itemsNearRoute(..., r.geometry, ...)` (l.1264) i `drawRoute(r.geometry)` (l.1267, `geometry.length < 2`) też rzucają na `undefined`, a brak `catch` → unhandled rejection. Bratnie fetchery są odporne: fetchTrafficForView (l.371, sprawdza status/`.catch`), loadFuelPrices (l.1214, `catch`), exportToFakturownia w invoices (l.502, `if (!res.ok)`) — plan() jest jedynym wyjątkiem.
- **Poprawka:** Po `const res = await fetch("/api/route", ...)` dodać strażnika: `if (!res.ok) { const e = await res.json().catch(() => ({})); toast(e.error ?? "Nie udało się wyznaczyć trasy.", "error"); return null; }`. Objąć całe ciało `plan()` blokiem `try/catch` (obok istniejącego `finally`), a docelowo — walidować kształt `RouteResponse` (obecność `geometry`/`segments`) przed użyciem.
